Session 1
The Developer's 10 Questions Before Coding

Ask these before writing a single line of code. These are not optional — they prevent mid-sprint surprises.

1. Layout
Overall structure? Responsive breakpoints? Grid system?
2. Components
What reusable components? All states? Existing reuse?
3. Content
Empty state? Very long content? Loading state?
4. Interaction
Clickable elements? Hover/focus/active? Animations?
5. Edge Cases
Error state per form? Slow network? Unauthorized user?
6. Accessibility
Tab order? ARIA labels? WCAG color contrast?
7. Data
Data needed from API? Refresh? Pagination?
8. Navigation
How to get here? Where to go? Back button?
9. Performance
Lazy loading? Code splitting? Caching?
10. Missing Specs
What's NOT in the design? What did designer assume?
Common Missing Specs
Often MissingWhat to Ask
Empty states"Page with no data?"
Loading states"While data is loading?"
Error states"When API fails?"
Long content"Title is 200 characters?"
Offline state"Without internet?"
Permission states"What does a viewer see?"
Animation specs"Duration and easing?"
Keyboard navigation"Focus order?"
Blockers vs Nice-to-Know

Classify each question before asking:

🚫 Blocker
Cannot code without answer
💡 Nice-to-Know
Can proceed, refine later

What Is a Design Review?

"A Design Review is a structured meeting where a team evaluates a proposed design — architecture, API, UI, or code — before implementation begins."

Review is NOT vs IS
❌ Finding fault
❌ Proving you're smarter
❌ Blocking progress
❌ Debating preferences
❌ A formality to check off
✅ Finding gaps
✅ Ensuring completeness
✅ Preventing rework
✅ Validating requirements
✅ Collaborative problem-solving
Cost of Changes by Phase
Design Phase
$1
Development
$5
Testing
$10
Production
$25+
Review Roles
✍️
Author
Presents the design, answers questions. The person or team who created the design.
🎯
Moderator
Keeps discussion on track, manages time. Usually tech lead or senior developer.
🔍
Reviewers (2–4)
Ask questions, raise concerns. Developer, QA, Product Owner, Designer — different lenses.
📝
Scribe
Documents decisions and action items. Any team member.
Review Ground Rules
  • Critique the design, not the person
  • Ask questions, don't make statements — "How would this handle 10k users?" not "This won't scale"
  • Bring alternatives, not just criticism
  • Focus on requirements, not preferences
  • Time-box discussions — if >5 min, take it offline
  • Document everything: decisions, action items, rationale

Session 2
Scope Lock 🔒 — Build Exactly What Was Agreed

"Scope Lock is the practice of freezing feature scope after design review — ensuring the team builds exactly what was agreed. Nothing more, nothing less."

Without Scope Lock
Day 1: "Let's build a login page"
Day 3: "Let's also add social login"
Day 5: "Let's add two-factor authentication"
Day 8: "Let's add biometric login"
Day 10: "Why isn't the login page done yet?"
With Scope Lock
Day 1: "Scope: email/password login + forgot password"
Day 3: "Social login is out of scope → ticket for Sprint 3"
Day 5: "Login page done, tested, deployed ✅"
Day 6: "Sprint 2 starts with next prioritized feature"
Sprint 1 — E-Commerce Login Example
✅
In Scope (must-have)
Email/password login · Forgot password · Session management · JWT tokens · Login form validation · "Remember me" checkbox · Error messages inline
🚫
Out of Scope (not in this sprint)
Social login (Google/Facebook) · Biometric authentication · Two-factor authentication · SSO/SAML · Magic link login
⏩
Deferred (future sprint — ticket created)
OAuth2 social login → Sprint 3 (ticket #147) · 2FA → Sprint 5 (ticket #201) · Biometric → Sprint 7 (ticket #289)
Handling Scope Change Requests

When "Can we also add social login?" comes in:

Is it critical for launch?
YES: estimate delay, team decides
Not critical?
Add to backlog → prioritize for future sprint

Session 2
Architecture Validation & Anti-Patterns

"Architecture that does not answer the problem statement is wrong, no matter how technically sound."

Common Architecture–Problem Mismatches
ProblemWrongRight
"Fast reads, rare writes"Write-optimized DBRead replicas, caching
"10 users, internal tool"Kubernetes, microservicesSimple monolith, SQLite
"Real-time collaboration"REST pollingWebSockets, CRDTs
"1M records nightly"Real-time streamingBatch processing, queue
"Simple CRUD app"Event sourcing, CQRSStandard MVC, PostgreSQL
YAGNI — You Aren't Gonna Need It

Don't build for imagined future requirements. When (IF) multi-tenancy is needed, refactor then — you'll have more information about the actual requirements.

Anti-Patterns in Design Review & Scope
🚲 Bikeshedding
Spending 30 min debating "should the button be blue or green?" while ignoring the missing error state. Fix: "Let's follow the design system. Moving on."
👥 Design by Committee
Everyone has a say, no one makes a decision — "let's discuss more next week" repeated for 4 weeks. Fix: "Tech lead decides by end of day. We build tomorrow."
✨ Gold-Plating
Requirements: "Users can upload a profile picture" → Implemented: AI-powered cropping, 12 filters, real-time optimization pipeline. Fix: JPEG/PNG, max 5MB, resize to 200×200, store in S3.
🧊 Analysis Paralysis
"We need to evaluate 15 databases before choosing one" → 3 months of evaluation, still no decision. Fix: "PostgreSQL meets all current requirements. We can migrate if needed. Let's build."

What Is Storybook?

"Storybook is a frontend workshop for building UI components and pages in isolation."

Without Storybook

❌ Developer: "I built it, please review"
Reviewer: Opens full app, navigates to page, struggles to trigger error state
Result: Edge cases never reviewed, bugs reach production

With Storybook

✅ Developer: "Here is the Storybook link, check all 6 states"
Reviewer: Opens Storybook, clicks: Default, Loading, Empty, Error, Overflow, Disabled
Result: Every state reviewable before a single API call

Storybook in the Lifecycle

← Design Review here

A Story = One Component in One State

Every component needs a story for every state. If a state is not in Storybook, it has not been reviewed. If it has not been reviewed, it will become a bug.

// OrderCard.stories.jsx
import { OrderCard } from './OrderCard';

export default {
  title: 'Features/Orders/OrderCard',
  component: OrderCard,
  tags: ['autodocs'], // auto-generates docs
};

// Each named export = one Story
export const Default = {
  args: { order: { id: 'ORD-001', customerName: 'Budi Santoso',
                     total: 250000, status: 'confirmed' } }
};
export const Processing = {
  args: { order: { ...Default.args.order, status: 'processing' } }
};
export const LongCustomerName = {
  name: '⚠️ Edge: Long Customer Name',
  args: { order: { ...Default.args.order,
                 customerName: 'Muhammad Rizky Fahrurrozi Abdurrahman' } }
};
// OrderList — Loading, Empty, Error states
export const Loading = { args: { orders: [], isLoading: true, error: null } };
export const Empty = { args: { orders: [], isLoading: false, error: null } };
export const Error = { args: { orders: [], isLoading: false,
  error: { message: 'Failed to load orders. Please try again.' } } };

FIFGROUP Loan Application System
📋 Sprint Goal

The PM hands you a wireframe for a new Loan Status Dashboard for Petugas Cabang. Your job this sprint: build the feature so branch officers can see real-time application status without calling HQ. Target: reduce 120 manual calls/day → <20.

🎯 What to Deliver

  • Design review findings — what is ambiguous, what is missing before coding starts?
  • Scope lock: IN (must ship), OUT (explicitly excluded), DEFERRED (future sprint)

What Makes a Strong Submission
✅Checklist questions are specific — "What happens when Scoring API returns 503?" not "What if there's an error?"
✅Scope lock answers the PM's real requests: push notif → OUT, CSV export → OUT, analytics → DEFERRED Sprint 3
✅Storybook stories name the actor: "Petugas Cabang sees APPROVED status for loan APP-2024-001"
✅Design review raises the branch isolation question — can Petugas Cabang JKT-001 see loans from JKT-002?
✅Architecture decision references the 120 calls/day → <20 target as the driver
✅Team can answer one unexpected design review question from the class
Common Mistakes to Avoid
  • Proposing WebSocket for 300 branches without discussing connection cost (YAGNI)
  • Scope lock that just says "notifications for later" — be explicit: which notification, which sprint
  • Storybook stories that only cover the happy path (APPROVED) — 5 states are required
  • Design review that misses security: Petugas Cabang must only see their own branch's loans

A scope lock isn't a "no" — it's a promise. When Petugas Cabang asks "can we see the full history?", a good scope lock says exactly when that will ship.
